## Findings

### Bugs auto-fixed

- **PTV toggle ruling documented** — `derro-shardcaller.md` Start-of-turn checklist item 4 now explicitly states Pack Tactics Voice is a toggle, not stackable ("multiple hits refresh duration but don't grant disadv on two saves"). Clarification only; no mechanical change.
- **CW stagger guidance added for shardcaller-only formations** — Stagger Barrages tactics note now explicitly says: when stagger is active in a shardcaller-only formation, the two non-barrage SCs should CW each other (not the barrage-firing SC). This is the formation where CW pays off — both multiattacking SCs get attack-roll advantage simultaneously. Low-risk authoring.

### Feel issues / DESIGN DECISIONS

**DD-37 (NEW): CW over-filtered in stagger context — 0/9 charges spent (2nd cycle)**
In this run (as in the 1st cycle), 0/9 CW charges were spent. This time the cause differs from the 1st cycle (where all 3 SCs front-loaded Barrage and CW was spent on Barrage turns): with the DD-16 stagger fix in place, SC1 fired Barrage, SC2 multiattacked, SC3 held. SC2's CW check filtered out SC3 as a target because `sc[s]["barrage_rdy"] = True` — the harness assumed Barrage-ready SCs would always Barrage rather than multiattack due to stagger. This is a harness simulation error, not a tactics text error. The tactics text (post this run's auto-fix) now correctly says non-barrage SCs should CW each other. In the real encounter, a DM following the stagger guidance would see SC2 CW SC3 before SC3's multiattack.
**Recommendation (harness):** Fix `run_phase_b.py` CW filter: track per-round action intent (did the SC already fire Barrage this round?) instead of filtering by `barrage_rdy` flag. Out of blast radius, logged for human infra fix.

**DD-18 (2nd cycle confirmed): Fireball trivializes shardcaller formation again**
Same result as 1st cycle: Marwen's L3 Fireball (24 fire, all fail saves) eliminates SC1+SC2 in R1. SC3 survives at 9 HP and falls R2 to Bazgar. Fight over in 2 rounds. Terrain constraint (columns limiting AOE to ≤2 targets) still unimplemented. Both cycles of this slice end in R2-R3 party victory via Fireball. DD-18 stands — human action required.

**FI-5 (NEW): Pack Tactics Voice zero-fire pattern — 2nd cycle**
PTV triggered 0 times in 2 rounds. Root cause: the only SC multiattack turn was SC2 (missed both throws); no other SC-initiated melee hit occurred. PTV's passive requires an ally HIT, which only happens on multiattack turns (Barrage doesn't produce hits). With front-loaded Barrage and a 2-round fight, the passive has zero surface area. PTV as a 3-SC-formation mechanic only activates if: (a) at least 2 SCs are multiattacking the same target on the same turn, AND (b) they both hit. In practice this requires R3+ and surviving SCs. DD-18 (Fireball trivializes) is the upstream cause — PTV cannot demonstrate in a 2-round fight.

### Mechanical questions

**INFRA: Roller output vs simulation damage mismatch**
The roller (`roll_action → roll_combat_action`) computes damage from `_mod._number_cache` (pre-populated integers modulated by dice size). The simulation's manual `_d(sides)` calls use a separate Python `random.Random` instance. These produce different values for the same roll:
- SC1 Barrage: roller output says 11 piercing; sim rolled 5.
- SC3 Barrage: roller output says 10 piercing; sim rolled 7.
At table, the roller output (printed verbatim) is what the DM uses — the sim damage values in this log are unreliable for absolute HP tracking. The narrative is correct; the exact damage numbers should be read from roller output, not sim. Out of cron blast radius — human to align sim damage with roller output by reading `result["rolls"]["damage_total"]` from the roller response dict.
