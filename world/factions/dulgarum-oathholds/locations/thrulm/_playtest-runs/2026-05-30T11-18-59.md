## Findings

### Bugs auto-fixed

- **BUG-SW18-A** — `shrine-touched-derro / oath_breaking_retaliation` DB `trigger.match` was `"took damage from a visible source"` (overly broad — includes ranged attacks, spells, area effects). The `.md` Reaction Priority section explicitly states OBR fires only against **"visible attacker within melee reach (5 ft)"** and specifically **"does NOT fire against ranged attackers, ranged-spell attackers, or area-effect damage."** Narrowed trigger description to `"took melee damage from a visible attacker within 5 ft"` via `upsert`. DB validates OK after fix. This is a documentation-level fix only (the DB `trigger.match` is descriptive, not mechanically enforced by the runner), but ensures a DM using the DB reference at the table applies the rule correctly.

### Feel issues / DESIGN DECISIONS

- **FI-SW18-A** — **Fireball altar-zone underwhelm (18th-cycle confirm).** Marwen burned a 3rd-level slot expecting fire-vulnerability doubling — STDs are `fire-vulnerable` per stat block. The altar zone suppressed the vulnerability, dealing normal 32 damage (half on save) instead of doubled. The fire dealt roughly the same as a standard L3 Fireball against a non-vulnerable target, making the choice feel wasteful. This is working **as designed** (the playtest note in the .md explicitly flags this), but there is no in-world cue that the altar zone is dampening the exploit. At the table a player casts Fireball expecting doubled damage and gets none — the runes should visibly char but not combust, or the DM should narrate "the fire washes over the runes but seems to dissipate into the stone." Consider adding a one-line narration cue to the `_overview.md` altar zone description to signal this to the DM for pre-combat player signposting. **Not a spec change — DM-coaching note.**

- **FI-SW18-B** — **Divine Smite radiant suppression (18th-cycle confirm).** Sabriel's 2d8=9 radiant smite was not doubled in the altar zone. Same as FI-SW18-A but for Sabriel's primary damage output. Combined effect: Marwen loses Fireball doubling + Sabriel loses smite doubling = the party's two highest burst-damage options are neutered in the altar zone. Party still won in 2 rounds because base damage was sufficient, but against higher-HP targets (or with worse dice) this suppression would dramatically extend encounters near the altar. Working as designed. Confirms the altar zone forces the party to drag enemies away or accept reduced damage.

- **FI-SW18-C** — **UF + OBR self-damage synergy confirmed.** std_b took 36 damage in R1 (16 fire from Fireball + 12+5 smite from Sabriel + 3 OBR psychic self-damage). The OBR self-damage counted correctly toward the 10-dmg UF threshold. UF gave advantage R2, producing a natural 20 crit that downed Sabriel. The mechanic chain worked elegantly: Sabriel hits hard → OBR fires, hurts itself → self-damage tips UF over threshold → advantage → crit → Sabriel downed. This is the UF mechanic working as intended (punishing the attacker who triggers OBR repeatedly). Feel: the crit felt earned, not random. The subtle mechanic (OBR self-damage → UF) is unlikely to be tracked at a real table without explicit DM attention. Consider adding a note in the UF checklist item: `"If OBR fired last round, add the 1d4 self-damage to the total."` — **this note already exists in the .md** (checklist item 4 explicitly calls this out). Confirmed the wording covers it.

### Mechanical questions

- **MQ-SW18-A** — **External random API blocked (18th consecutive cycle).** Both `quantumnumbers.anu.edu.au` and `random.org` return errors in this sandbox environment. Phase A required pre-seeding `dnd_roller._number_cache` with Python `random` before calling `roll_combat_action`. All 29 actions passed once pre-seeded. The workaround is valid but fragile (must refill the cache if it empties mid-run). **Infrastructure note for the human:** `dnd_roller._ensure_numbers()` has no local-Python fallback path — both remote APIs are exhausted and the function raises `RuntimeError`. A 3-line addition (`import random as _pyrand; return [_pyrand.randint(1, 65535) for _ in range(needed)]` as last-resort fallback) would make the cron self-healing on air-gapped or rate-limited deploys. Not urgent — the cache pre-seed works — but worth addressing.

- **MQ-SW18-B** — **AR fired before Fireball, both melee PCs saved (~16% probability).** STD-B won initiative (22) and fired AR before Marwen's Fireball (init 21). Bazgar (save 18 vs DC14) and Sabriel (save 15 vs DC14) both passed. P(both save) ≈ (8/20)² ≈ 16%. If either had failed, they'd have taken 10 necrotic + 1d4 psychic, meaningfully changing the subsequent fight dynamics (Sabriel at ~25 HP entering R2 would have died on Atk1 rather than Atk2 of the UF multiattack). The simulation outcome was slightly favored by dice. Not a design concern — just noting that the initiative spread (STD-B 22, Marwen 21) means STD-B *almost always* fires AR before Fireball, and the save result determines the round-1 texture. If both PCs fail R1 AR, the party HP budget is significantly lower heading into R2.

- **MQ-SW18-C (minor, not previously logged)** — **AR narration output says `(ASKING PLAYER)` for saves.** This is correct (players roll their own saves at the table), but in a solo-run simulator context the phrasing is confusing — the harness appears to "ask" an absent player. Low priority; no spec change needed, just a note for playtest-bot runners.
