**HP math:** glacier-stalker max HP = 84 (8d10+40 confirmed from stat file). 84 − 50 = 34. `assert_hp: 34` ✓

**Bloodied threshold:** 50% of 84 = 42. At 34 HP, glacier-stalker is bloodied. ✓

**`expect_watch` structural issue — SCEN-R240-01:**

The `cure_wounds` DB entry for `aelric-frostweaver`:
```json
"watch": {"event": "bloodied", "scope": "ally", "priority": 20}
```

The scenario brings the **enemy** NPC (glacier-stalker) to bloodied, then asserts `expect_watch(aelric-frostweaver, cure_wounds)`. With `scope: "ally"`, the watch should only fire when an ally of Aelric's team crosses the bloodied threshold — not when the enemy does.

Two interpretations:
- **A (scope works correctly):** `cure_wounds` does NOT fire when glacier-stalker is bloodied; the test FAILS; `max_llm_fallback: 0` prevents the LLM from masking the failure. Scenario is a regression gate for a future scope-bug.
- **B (scope is broken):** `cure_wounds` fires on any bloodied event regardless of scope; the test PASSES; the pass represents a bug, not working behavior.

The scenario title says "verify cure_wounds suggestion fires on Aelric" — framing A is correct only if the title is wrong, framing B is a test designed to document a known scope failure. Most likely interpretation given `max_llm_fallback: 0`: **this scenario is expected to FAIL at current implementation**, demonstrating the scope-filtering bug. The test is a regression gate: it SHOULD fail until scope filtering is implemented correctly.

No equivalent mountin-pass scenario note in the open-issue tracker from previous fires. Log as SCEN-R240-01 below.

**Structural notes:**
- `max_llm_fallback: 0` ✓ — correctly regression-gated.
- No `{kind: click_round}` step — only tests single-hit state transition; does not verify persistence across rounds.
