## Phase B — Generative Playtest: `gnoll-pack-segments`

*scenarios.yml index 1 of 20 total. Counter 241 → 242. This is a mountin-pass scenario (encounter: mountin-pass); full thrulm dice-math encounter simulation is out of scope. Structural review follows.*

**Scenario structure:**

```yaml
turns:
  - {kind: switch_tab, slug: gnoll-pack}
  - {kind: type, tab: gnoll-pack, text: "-22"}        # kill m3
  - {kind: type, tab: gnoll-pack, text: "-15"}        # m2 to 7
  - {kind: type, tab: gnoll-pack, text: "m1 -22"}     # kill m1
  - {kind: assert_member_hp, slug: gnoll-pack, expected: [0, 7, 0]}
targets:
  max_click_count: 5
```

**HP math:** Assuming gnoll-pack members each have 22 HP (standard MM gnoll, 5d8, avg 22):
- Turn 1: `−22` → drains m3 to 0 (right-to-left auto-drain). Members: [22, 22, 0]
- Turn 2: `−15` → drains m2 to 7. Members: [22, 7, 0]
- Turn 3: `m1 −22` → explicit targeting kills m1. Members: [0, 7, 0]
- `assert_member_hp: [0, 7, 0]` ✓ — math is self-consistent at 22 HP per member.

**Step count vs max_click_count:** 4 turn steps (`switch_tab` + 3 types + 1 assert). `max_click_count: 5` is consistent.

**What this scenario gates:**
1. Mob segment drain order is right-to-left (m3 first, m2 second).
2. Explicit `m<n>` targeting overrides auto-drain order.
3. `assert_member_hp` (array form) correctly verifies per-member HP state.

**SCEN-R241-01 — missing `max_llm_fallback: 0`:**

`gnoll-pack-segments` does not set `max_llm_fallback`. Without this gate, if mob drain order is wrong (e.g., draining left-to-right, killing m1 first), the LLM can narrate the outcome in a way that makes `assert_member_hp` succeed by reinterpreting which segment died. The drain-order assertion is the core mechanic being tested — `max_llm_fallback: 0` should be added to prevent LLM papering. See Phase C log.

**Thrulm relevance note:** The gnoll-pack mob segmentation mechanic is the same code path used by any future thrulm mob (if multiple thralls or ragers are grouped). The thrulm encounter currently has no mob NPCs, but if a "thrall mob" (2–3 thralls as a single mob token) is ever added, this scenario validates the infrastructure they'd rely on.
