### Simulation — stalker-solo-bloodied

**Turn 1 simulation:**
- Input: `-50` to glacier-stalker tab
- Result: 84 − 50 = **34 HP** ← below bloodied threshold of 42 ✓
- `assert_hp: expected 34` → **PASS** (arithmetic correct)

**Watch trigger question:**
- `expect_watch, slug: aelric-frostweaver, action_name: cure_wounds`
- Aelric's `cure_wounds` watch has `scope: "ally"` — fires when an ally enters bloodied
- Glacier-stalker is an **enemy**, not Aelric's ally
- **Expected result: watch should NOT fire based on scope semantics**

---

### Findings

**SCENARIO-BUG-R120-01 (new):**  
`stalker-solo-bloodied` scenario expects `aelric-frostweaver`'s `cure_wounds` watch to fire when the **enemy** glacier-stalker goes bloodied. The watch has `scope: "ally"` — by design it should only fire for friendly creatures. The `expect_watch` assertion will either:

- **FAIL** (correct behaviour): scope filter enforced, watch doesn't fire on enemy-bloodied — scenario assertion is wrong
- **PASS for wrong reason**: scope filter NOT enforced, watch fires on any bloodied event regardless of scope — scenario passes but masks a latent bug where Aelric would offer to heal enemies

Recommend one of:
(a) Fix scenario spec: replace `-50` on glacier-stalker with damage to an allied PC tab to correctly test ally-bloodied → cure_wounds surface
(b) If intent is that bloodied events are unscoped in the GUI (any creature's bloodied fires all bloodied-watches), update Aelric's DB entry from `scope: "ally"` to `scope: "any"` and document the design choice

**SCENARIO-BUG-R120-02 (new):**  
If the GUI's bloodied-event watch does NOT filter by scope (causing the scenario to PASS), this is a latent bug: Aelric would surface `cure_wounds` every time ANY creature (including the beholder, ragers, etc.) drops below half HP. In a multi-combatant encounter, this generates false-positive cure prompts on enemies. The `scope` field in the DB watch definition is meaningless if not enforced. Recommend scope filter audit across all `watch` entries in actions.jsonl.
