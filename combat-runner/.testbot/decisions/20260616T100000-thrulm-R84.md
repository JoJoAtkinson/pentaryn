### NEW Phase A Finding This Fire

**NEW-R84-01 (LOW — LOG ONLY): Verb ambiguity between `disintegration_ray` and `void_ray`**

`disintegration_ray` verbs: `["disintegrate", "disintegration", "ray", "force ray", "beam"]`
`void_ray` verbs: `["void ray", "void", "force ray", "ray (legendary)"]`

Both DB rows contain `"force ray"` as a verb. If a DM types "force ray" in the runner, the action dispatch is ambiguous — the runner may fire either ability. In practice, "disintegrate" and "void ray" are the natural invocations, so this is unlikely to trigger in normal play. However, the shared entry is a latent misfire risk if a DM unfamiliar with the encounter improvises. Safe fix: remove `"force ray"` from `disintegration_ray`'s verbs (leaving `"disintegrate"`, `"disintegration"`, `"ray"`, `"beam"` as unambiguous alternatives). Requires `combat_action_upsert` + validate. NOT auto-fixed here.

## Phase B — Scenario Audit: `thrulm-disintegration-slot-consumed` (scenarios.yml index 33)

**Scenario:**
```yaml
id: thrulm-disintegration-slot-consumed
title: "Thrulm: Disintegration Ray — recharge slot consumed after firing"

  - {kind: switch_tab, slug: beholder-thrulm}
  - {kind: type, tab: beholder-thrulm, text: "2 disintegrate"}
  - {kind: assert_slot, slug: beholder-thrulm, action: disintegration_ray, remaining: 0}
```

**Verb resolution check:**
- `"2 disintegrate"`: combatant 2 = Marwen (id "2"). Verb "disintegrate" → `disintegration_ray` verbs include "disintegrate". ✓
- No ambiguity with `void_ray` (which has "void", "void ray", "force ray", "ray (legendary)"). "disintegrate" is unambiguous. ✓

**Recharge slot math:**
- DR starts recharged at combat start (`recharge: 5`, initial remaining: 1).
- After firing: remaining = 0.
- `assert_slot: remaining: 0` → should PASS. ✓

**Mechanical correctness:**
The scenario is sound. It validates recharge slot consumption after a named ability fires. The grammar routing `<target_id> <verb>` = `2 disintegrate` fires DR at Marwen, consuming the slot.

### What the scenario does NOT cover (FEEL-R84-01 — LOG ONLY)

The test validates the recharge mechanic but does not assert:
1. **HP reduction:** No `assert_hp` for Marwen after DR fires. If the damage pipeline has a bug (wrong formula, wrong target, no damage applied), the test still passes as long as the slot is consumed.
2. **Disintegration rider:** No assertion that the "turned to ash" trigger fires when HP reaches 0. This is the most dramatic effect of DR (no revivify); it is completely untested by the scenario.
3. **Hit-or-miss branch:** The scenario assumes a hit (verb dispatch succeeds and damage is applied). If the runner simulates a miss, does the slot still consume? This edge case is untested.

**Recommendation (LOG ONLY):** Add a complementary scenario `thrulm-disintegration-hp-assert` that pre-sets Marwen to 1 HP (via `"-31"` input) and then fires DR, asserting `expected: 0` HP and optionally checking for a disintegration log line. This would close the damage-pipeline gap without the runner needing auto-condition support.

### Feel Analysis — DR in the context of this party

- Marwen (32 HP, AC 15): +7 to hit (with Void-Feeding), needs 8+ = 65% hit rate.
- On hit: 10d8 force, avg 45. Any value ≥33 downs Marwen; P(10d8 ≥ 33) ≈ 97%. P(Marwen disintegrated) ≈ 0.65 × 0.97 ≈ **63%** per use.
- At recharge 5–6 (33% per turn): expected 3 turns between uses. Against a 3-PC party with 3 HP pools, the beholder can permanently eliminate one caster roughly every other combat (depending on rest cycles).
- **No save.** DR is a pure attack roll — no DEX save, no Legendary Resistance interaction from the party's side. A gust-of-wind repositioning or Wall of Stone blocking line of sight is the only counterplay. Neither is available at L5 without Marwen's spell slots (which are being drained by DD). This creates a hard dependency loop: DD strips Marwen's slots → Marwen can't reposition → DR kills Marwen → no revivify.

This loop was previously noted (FEEL-F80-01, FEEL-R83-01) but the dependency chain from DR through DD to slot exhaustion is now confirmed through this scenario's lens.

### FEEL-R84-02 (NEW — LOG ONLY): threshold-patrol provides zero resource attrition before the shrine

**Issue:** The patrol (2× DW + 1× Thrall, ~150 XP) ends in Round 1 with negligible party costs. Marwen spends a single L2 slot; Sabriel spends a smite slot; Bazgar uses Action Surge (refreshes on short rest). HP pools are untouched. The party enters the beholder encounter effectively at full strength.

**Design intent question:** Is threshold-patrol meant to:
- (a) **Create resource attrition** before the shrine (narrative purpose)? If yes, the composition fails — 2 DW + 1 Thrall cannot crack AC 18/19 or survive a single round of L5 offense.
- (b) **Signal danger / worldbuilding** (players learn the guards are corrupt/thralled) without mechanical pressure? If yes, the composition may be intentional, but this should be explicitly noted in the encounter docs.

**If (a) is intended**, recommended minimum patrol: add 1 Shardcaller + 1 Rager. Adjusted XP: 25+25+50+450+450 ≈ 1,000 XP (2-enemy multiplier for 4 NPCs: ×2 → ~2,000 effective). That's over Deadly — too many. Better option: 2× DW + 2× Thrall + 1 Shrine-Touched = 25+25+50+50+700 ≈ 850 XP × 2 multiplier = 1,700 — still over. Simpler: add 1 Shardcaller only: 25+25+50+450 = 550 XP × 1.5 (4 NPCs) = 825 XP ≈ Hard. This creates meaningful SC pressure (Call Weakness + Barrage), strains Marwen (SC fires first, Barrage threatens formation), and costs 1-2 L2 slots. Realistic attrition without a TPK.

**Beholder DD dependency note:** If threshold-patrol is expanded to drain even 1 L2 spell slot from each caster, the beholder DD loop becomes one round faster — Sabriel has one fewer slot to strip. This changes the shrine encounter from "party loses R3–R4" to "party loses R2–R3." With the "meant to lose" design intent, this tightens the experience but doesn't change the outcome.
