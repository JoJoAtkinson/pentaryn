## Phase B — Generative Playtest: `matron-slots`

*scenarios.yml index 2 of 20 total. Counter 242 → 243. This is a gar-vally scenario (encounter: gar-vally); full thrulm dice-math encounter simulation is out of scope. Structural review follows.*

**Scenario structure:**

```yaml
- id: matron-slots
  title: Matron uses moonbeam — slot decrements, chip greys out
  encounter: gar-vally
  prelude: |
    The matron rises from the fire-pit. Three runners flank her.
  turns:
    - {kind: switch_tab, slug: matron-of-the-valley}
    - {kind: type, tab: matron-of-the-valley, text: "moon"}
    - {kind: assert_slot, slug: matron-of-the-valley, action: moonbeam, remaining: 0}
  targets:
    max_click_count: 5
```

**What this scenario gates:**
1. Short verb "moon" fuzzy-resolves to the `moonbeam` action (verb-matching, not LLM).
2. After one cast, moonbeam slot count decrements to 0 (i.e., 1/day usage).
3. UI chip for moonbeam visually greys out (title says "chip greys out"; assert_slot verifies the backing state).

**Step count vs max_click_count:** 2 active operations (switch_tab = ~1 click, type = ~1 click; assert_slot is a verification, not a click). `max_click_count: 5` gives 3 clicks of headroom — reasonable for possible tab-load or confirmation clicks. ✓

**SCEN-R242-01 — missing `max_llm_fallback: 0` and undocumented slot-count assumption:**

`matron-slots` does not set `max_llm_fallback`. Two problems:
1. The verb "moon" is 4 characters — shortest verb in any thrulm/gar-vally/mountin-pass scenario reviewed. If it doesn't match exactly, the LLM can narrate "the matron begins to cast moonbeam..." and the `assert_slot` check could succeed against a narration-only event rather than an actual slot decrement. `max_llm_fallback: 0` should be added.
2. `assert_slot remaining: 0` assumes the matron has exactly **1** moonbeam charge. If her DB row grants 2+/day (e.g., `slots: 2`), the assertion fails on correct behavior (remaining would be 1, not 0). The assumed slot count is not documented anywhere in the scenario. Recommend adding a comment: `# assumes moonbeam is 1/day per DB`.

**Thrulm relevance note:** The slot-decrement mechanic exercised here is the same system that tracks PC spell-slot consumption after a Drain Divinity (beholder strips the highest slot from Sabriel or Marwen, runner must reflect the decrement on the PC tab). If slot counts drift or don't decrement correctly, Drain Divinity's economy becomes untestable. This scenario is an indirect regression gate for Drain Divinity tracking in the thrulm encounter.
