### Scenario-Specific Observations

**Test (1) — Multiattack sub-attacks with grapple riders:**

The multiattack correctly fires all 3 sub-attacks per the .md spec (2× Tentacle Lash + Maw). Grapple rider fires on each Tentacle hit. Maw rider (escape disadvantage) requires DM manual tracking — no DB representation. **No DB gap for grapple rider on Tentacle** (it's implicit in the single_attack spec; the runner narrates it). However, the escape DC (16) is not stored in the DB row for `tentacle_legendary` or the multiattack Tentacle sub-attacks — DM must know it from the .md. Minor gap; the multiattack narration doesn't surface the escape DC verbally.

**Test (2) — "lash" routes to `tentacle_legendary`:**

DB confirmed: `tentacle_legendary` verbs = ["legendary tentacle", "lash", "reach"]. "lash" IS in this list. Typing "lash" correctly fires tentacle_legendary.

However, VERB-UX-01 (OPEN): "tentacle" is in the `multiattack` verb set and NOT in `tentacle_legendary`. A DM who types "tentacle" to spend a Legendary Action will fire the full multiattack instead. In live play this is a real trap — the cheat sheet says the LA is called "Tentacle" (not "lash"), so the natural DM input is "tentacle." The correct verb "lash" is undocumented in the cheat sheet's verb legend.

**Test (3) — Antireality timing regression (R224):**

Antireality trigger: `event: "action_executed"`. The scenario step is:
1. type "lash" (beholder's LA tentacle attack)
2. type "-15" (simulate PC hitting beholder for 15 damage)
3. assert_hp = 95

The Antireality reaction should fire when an incoming ATTACK is declared against the beholder — not when raw "-15" damage is typed. If the runner correctly distinguishes `action_executed` (an attack action being announced against the beholder) from a raw HP delta input ("-15"), then:
- "-15" → no Antireality prompt
- An attack-type input from a PC tab directed at the beholder → Antireality prompts

In the headless test, "-15" is typed on the *beholder's own tab*, not a PC tab. The runner should not interpret this as an incoming attack event. If it does, the R224 fix didn't hold. The assert_hp = 95 would still pass (the damage applies regardless of whether Antireality prompts), but the DM experience breaks — Antireality would pop unnecessarily on self-damage inputs or trap damage.

**No new crash-level bug found here** — this is a UX quality check that requires GUI observation to fully verify. Logging as FEEL-R237-01 below.

**Test (4) — assert_hp = 95 (110 - 15 = 95):**

If beholder starts at full HP (110) and takes 15 damage from a PC attack, expected HP = 95. THP from Drain Divinity could absorb part of this. In round 1 the beholder hadn't used Drain Divinity yet (no 3 LA spent until end of Bazgar's turn), so at the point of the PC's attack in the scenario test the beholder should be at full 110 HP with no THP. 110 - 15 = **95 ✓**.

### Legendary Action Economy Confirmed

Across 2 simulated rounds:

- **Drain Divinity (3 LA):** Fired twice. Round 1: worked (Sabriel failed CHA save, lost 3rd-level slot, beholder gained 6 THP). Round 2: failed (Sabriel succeeded save, wasted all 3 LA). Average expected THP gain per Drain Divinity use vs Sabriel (CHA +5): ~30% success for Sabriel → beholder gets THP ~70% of rounds. EV: 0.7 × 6 THP = 4.2 THP per round. This is confirmed lower than Void Ray EV (FEEL-R224-03).

- **Dominant pattern:** Drain Divinity every round when Sabriel is alive. In rounds where Sabriel makes her save, the beholder wasted its entire legendary budget. This is the "degenerate every-round" pattern already flagged — and the round where it fails is catastrophic for the beholder (zero legendary actions effective). The RNG variance is high.

- **Move (1 LA):** Never used in 2 simulated rounds. The beholder was already at combat range. Consistent with prior observations (Move is used only on retreat).

- **Void Ray (2 LA):** Never used (Drain Divinity consumed all 3 LA). Only available if Sabriel is dead or her slots are exhausted. Then Void Ray + Tentacle = 3 LA total.
